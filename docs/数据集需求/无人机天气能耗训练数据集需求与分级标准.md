# 无人机天气能耗训练数据集需求与分级标准

## 1. 文档目的

本文档用于指导后续无人机数据集收集、整理、筛选与分级。

当前项目目标不是做飞控，也不是复原飞机内部全部控制细节，而是构建一个可部署的“受天气影响的无人机能耗预测核心模块”。

这意味着数据集选择必须满足两个条件：

- 能支撑训练真实能耗预测关系。
- 训练时使用的主要输入，在部署时也能够被系统生成或获取。

因此，数据集不只要看“字段多不多”，还要看“字段语义是否清楚”“部署前能否拿到”“缺失后能否补回”。

## 2. 当前已确认的工程约定

这些约定用于后续数据集筛选、字段审计、特征转换和部署接口设计。

- 任务输入航速默认采用地速。
地速指无人机相对地面的计划速度，适合作为多旋翼航点任务输入。内部字段命名为 `planned_ground_speed_mps`；旧字段 `speed_mps` 暂时只作为兼容别名。模型内部可以继续保留环境风速、风向、逆风、侧风，也可以由地速和风场进一步计算等效空速或等效相对风。

- 机载风速、机载风向不作为飞行前部署主输入。
原因是起飞前无法直接获得未来飞行过程中的机载风速和机载风向。它们更适合用于校准和验证“天气风场 + 计划航线 + 计划地速 -> 机载等效相对风”的转换模块。

- 高度字段至少保留两类口径。
一类是海拔高度，用于天气、气压、空气密度和高度层风场对齐；另一类是离地/超障相关高度，用于城市高楼环境下的安全余度、建筑绕障和航线走廊约束。

- 当前主对象是多旋翼无人机。
数据结构可以预留机型字段，但当前训练和误差分析不为固定翼、复合翼或 eVTOL 额外设计能耗机理。

- 输出粒度同时保留任务级和分段级。
任务级结果用于判断整条任务能否完成，分段级结果用于定位高风险航段，并支撑后续路线、高度和航速优化。

## 3. 训练阶段不能大量依赖的字段

下面这些字段不是完全不能看，而是不能作为默认主输入大量依赖。它们更适合做标签、审计字段、分析字段或辅助对照字段。

- 飞行过程中机载实时测得的风速、风向、相对风估计。
原因：部署前通常拿不到同口径数据。真实项目应使用天气预报、历史天气、局地风场估计、任务前测风、计划航线和计划地速生成等效风特征；机载风字段优先用于校准和验证这个转换过程。

- 实飞逐时刻高度抖动序列。
原因：这类数据包含传感器噪声、气压/GNSS 抖动、控制振荡，训练时容易学到“飞后才知道”的细节，部署前无法获得。

- 飞完之后才能确定的真实轨迹修正结果。
例如绕障后的实际偏航、小范围漂移、局部悬停抖动。
原因：任务前只能拿到计划路线，拿不到飞完后的真实控制过程。

- 直接贴近目标值的输入。
例如当目标是预测功率或能耗时，再把同段 `mean_power`、`segment_energy`、`battery_current` 作为主输入。
原因：这等于把答案或接近答案的量塞回模型，离线分数会虚高。

- 数据集来源标签。
例如 `source_dataset`、`source_is_xxx`、`paper_id` 之类。
原因：这会让模型学习数据集差异，而不是学习物理规律。

- 语义不清的重量、高度、风字段。
例如不知道是载荷重量还是总重，不知道是相对高度还是海拔高度，不知道是环境风还是机载相对风。
原因：语义错了，模型学到的是伪规律。

一句话说，训练可以使用“部署前可得输入 + 可由规划器和天气转换模块生成的派生特征”，不能把“飞行后日志细节”误当作部署输入。

## 4. 当前项目需要的数据集长什么样

最理想的数据集应当描述一次完整任务，而不是孤立的一条传感器曲线。

一条高价值样本最好能覆盖：

- 任务开始时间。
- 飞行轨迹或可恢复轨迹。
- 速度与高度变化。
- 电池/功率/能耗标签。
- 机型、载荷、电池等任务常量。
- 能与天气时空对齐的定位和时间信息。
- 最好还能区分起飞爬升、中段巡航、末段下降这类任务结构。

对城市高楼环境而言，额外重要的是：

- 转弯多不多。
- 是否存在局部绕障。
- 是否有 3D 航点或楼宇环境约束。
- 天气风场是否只有粗网格背景，还是包含近地局地风信息。

## 5. 字段分层

### 5.1 一级：所有候选数据集都应该记录的描述字段

即使这个数据集暂时不能训练，也必须先把这些说明信息记录下来。

- 数据集名称。
- 来源链接。
- 对应论文或项目。
- 许可协议。
- 真实飞行还是仿真。
- 机型名称。
- 场景类型。
例如城市、高楼、郊区、海边、山地、室内。
- 原始文件类型。
例如 CSV、ulog、bag、mat、zip。
- 采样率或时间分辨率。
- 是否有字段说明文档。
- 是否有单位说明。
- 是否有字段语义说明。
例如风速是环境风还是机载估计风。
- 速度字段口径是否清楚。
例如任务地速、GPS 地速、空速或机载相对风相关速度。

如果连这些说明都没有，这个数据集也可以收集，但默认只能先归为低等级待审计数据。

### 5.2 二级：进入主训练闭环的必须字段

这是“真飞行能耗训练主集”的核心门槛。

- 时间戳。
至少要能确定每个样本点发生在什么时间。

- 位置。
至少要有经纬度，最好还要有高度。

- 飞行持续时间或连续时序。
不能只有任务总量，最好能看到飞行过程。

- 能耗标签。
至少满足下面之一：
`电压 + 电流`、`功率`、`分段能量`、`总能量`。

- 距离或可恢复距离。
如果没有直接距离，至少要能从轨迹恢复。

- 速度或可恢复速度。
如果没有直接速度，至少要能从轨迹和时间恢复。
当前优先接受地速口径；如果数据集给的是空速或机载速度，必须标记清楚，不能直接和地速混用。

- 高度信息。
哪怕语义暂不清楚，也必须有字段或文档线索，后续才能做相对高度/海拔高度审计。

- 任务时间与位置足以回填天气。
如果没有直接天气字段，至少要能基于时间和位置去查历史天气。

### 5.3 三级：强烈推荐字段

这些字段缺了不一定不能用，但有了会大幅提高训练价值。

- 载荷重量。
- 起飞总重。
- 空机重量或机型规格。
- 电池容量、串数、额定电压。
- 风速、风向、阵风。
- 温度、湿度、气压、降水。
- 任务计划高度。
- 任务巡航速度或分段计划地速。
- 航点列表或分段起终点高度。
- 飞行模式或任务阶段标签。

### 5.4 四级：可选增强字段

- 电机转速、桨盘信息、推进系统配置。
- 姿态角、角速度、加速度。
- 航向角。
- 机载测风与环境测风同时存在的对照字段。
这类字段尤其适合校准“天气风场 + 计划航线 + 计划地速 -> 机载等效相对风”的转换模块。
- 电池健康度、循环次数、内阻。
- 城市障碍物模型、楼宇高度、地图约束。

## 6. 字段可恢复性分级

这个分级非常重要，因为你接下来会收集很多“不完整”数据集。

### 6.1 容易恢复的字段

- 航向。
可由连续经纬度点恢复。

- 距离。
可由轨迹积分恢复。

- 平均水平速度。
可由距离和时间恢复。
默认解释为地速；如果数据集文档明确是空速，需要单独标记。

- 分段能量。
可由 `电压 * 电流 * 时间` 计算。

- 平均功率。
可由能量和时长计算。

- 天气背景字段。
可由 `时间 + 经纬度 + 高度近似` 回填历史天气或预报天气。

- 逆风、侧风。
可由 `环境风 + 航向` 计算。

### 6.2 勉强可恢复，但风险较高的字段

- 计划高度口径。
如果只有任务常量高度而无航点高度，能做 `planned_level`，但不能真实表达 3D 路径。

- 分段爬升/下降特征。
如果只有起终点高度，可以做保守 `route_3d` 近似，但不能替代真实完整 3D 轨迹规划器。

- 质量字段。
有时可根据文档、常量偏差、机型规格推测“载荷”还是“总重”，但风险较高。

### 6.3 很难恢复的字段

- 真实环境风与机载相对风的精确区分。
- 高度字段到底是海拔高度还是相对起飞点高度。
- 电池老化状态。
- 真实机体气动构型差异。
- 飞行中临时绕障后的真实控制细节。

这些字段如果原始数据集没给清楚，就必须降级处理，不能假装已经恢复成功。

## 7. 数据集分类

同一个数据集可以同时有“类型”和“等级”两个标签。

### 7.1 类型分类

- `T1 主闭环飞行集`
真实飞行，具备轨迹、时间、能耗标签，适合直接进入主训练链。

- `T2 可天气回填飞行集`
真实飞行，具备轨迹、时间、能耗标签，但原始天气不足，可通过外部天气回填。

- `T3 天气/风场增强集`
有飞行与天气关系，但能耗标签弱，适合作为风场机理、字段语义、阶段规律增强集。

- `T4 电池/动力系统机理集`
台架、电池放电、推进系统实验数据。
不直接训练任务级能耗，但可补充机理约束。

- `T5 城市 3D 路线/障碍环境集`
重点在 3D 路径、建筑约束、城市地图，不一定有能耗标签。
适合后续路线规划与 3D 特征生成模块。

- `T6 仿真/参考集`
仿真、论文附件、小样例、补充表格。
适合参考，不直接当主训练集。

### 7.2 等级分级

- `A级`
可直接进入主训练候选。
通常满足：真实飞行 + 时间 + 轨迹 + 能耗标签 + 机型/载荷/高度至少部分清晰。

- `B级`
少量适配后可以进入主训练。
通常天气需回填，或质量/高度语义需要审计。

- `C级`
不能直接主训练，但能做增强或辅助验证。
例如有风场和轨迹，但能耗标签不足。

- `D级`
只能做参考或局部模块支撑。
例如只有电池放电、只有城市地图、只有天气。

- `E级`
信息严重不全、仅截图/表格、语义不清。
先收集，不直接投入训练。

## 8. 主训练最小门槛

如果一个数据集要进入当前主训练链，最低建议满足下面条件：

- `A级或B级`。
- 类型属于 `T1` 或 `T2`。
- 有时间戳。
- 有经纬度轨迹。
- 有可计算能耗标签。
- 有距离和时长。
- 有速度字段或可恢复速度。
- 有高度字段或计划高度字段。
- 至少能回填天气背景。

如果缺少其中一项，也可以收集，但先不要误放进主训练集。

## 9. 不满足必须字段时，应该怎么收集

即使数据集不满足主训练门槛，也建议继续收集，但要分箱整理。

- 缺天气但有时间和轨迹。
照样收，归 `T2`，后续回填天气。

- 缺能耗标签但有天气和轨迹。
照样收，归 `T3`，用于风场/阶段分析。

- 缺轨迹但有任务总能量和任务配置。
照样收，归 `C级` 或 `D级`，用于任务级统计，不用于分段训练。

- 只有电池/电机/推进实验。
照样收，归 `T4`。

- 只有城市 3D 地图、障碍物或航线规划样例。
照样收，归 `T5`。

## 10. 数据集登记表建议字段

建议你后续整理数据集时，每个数据集至少登记下面这些字段：

- 数据集名称
- 类型
- 等级
- 真实飞行/仿真
- 城市场景/非城市场景
- 是否有时间戳
- 是否有轨迹
- 是否有高度
- 是否有能耗标签
- 是否有天气字段
- 是否可回填天气
- 速度字段口径是否清楚
- 是否有载荷/重量
- 是否有机型规格
- 风字段语义是否清楚
- 高度字段语义是否清楚
- 是否有字段说明文档
- 当前是否可进入主训练
- 当前主要用途

## 11. 当前阶段最值得优先找的数据集

最优先的是：

- `T1/A级`
真实飞行 + 轨迹 + 时间 + 电压/电流/功率 + 高度 + 机型/载荷信息。

第二优先的是：

- `T2/B级`
真实飞行 + 轨迹 + 时间 + 能耗标签，但天气需要后补。

第三优先的是：

- `T5/C级`
有城市 3D 航点、楼宇约束、路线结构信息的数据集。
它们能帮助后续把 `route_3d` 继续升级成更真实的任务规划输入。

## 12. 当前推荐判断原则

一个数据集不是“字段越多越好”，而是要先问三个问题：

- 这个数据集能不能提供真实能耗标签。
- 这个数据集里的主要输入，在部署前能不能拿到或生成。
- 这个数据集缺的字段，后续到底能不能可靠补回。

如果这三个问题里前两个都答不好，这个数据集就不应该进入主训练，只能做辅助。

## 13. 一句话结论

当前主线应优先收集“真实飞行 + 轨迹 + 时间 + 能耗标签 + 可回填天气”的数据集；同时保留城市 3D 路线、局地风场、电池机理等不完整数据集，按类型和等级分箱管理，不要混成一个大杂烩训练集。

# 共享环境策略

## 当前决策

仓库内的 Python 运行环境集中放在：

```text
fibocom_slz/._envs/
```

当前第一套共享环境：

```text
fibocom_slz/._envs/uav-modules-py310
```

## 管理原则

- 尽量复用共享环境，不按“子目录数量”拆环境。
- 只有出现明显冲突时才创建并行环境。
- 提交到版本库的是环境描述和 bootstrap 逻辑，不是环境目录本身。
- 统一通过 `tools/env-bootstrap` 进入、更新和清理环境。

## 拆分判断标准

适合复用同一环境：

- Python 版本一致
- 第三方依赖重合度高
- 不涉及互斥的 GPU / 仿真 / 原生库栈

适合新建并行环境：

- Python 版本不同
- 一部分模块需要重型仿真运行时
- 一部分模块只需要轻量数据处理依赖
- 原生库冲突使共享维护成本过高

## 目录职责

`._envs/`

- 真实环境前缀
- 不提交环境内容

`tools/env-bootstrap/manifests/`

- 环境入口清单
- 描述哪个模块组复用哪套环境

`modules/*/environment.yml`

- 模块组依赖声明
- 用于构建环境，而不是直接承载环境本体

## 当前 UAV 例子

以下两个模块当前共用一套 `uav-modules-py310`：

- `modules/uav/uav-weather-energy-predictor`
- `modules/uav/uav-gis-replay-studio`

原因：

- 都是 Python 3.10
- 依赖高度重合
- 都属于轻量数据/可视化链路
- 目前没有 ROS / CUDA / Gazebo 级别冲突

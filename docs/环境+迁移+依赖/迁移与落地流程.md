# 迁移与落地流程

## 目标

当一个模块从旧工程迁移到当前仓库时，需要同时处理三件事：

1. 代码落位
2. 依赖收敛
3. 环境归属

## 推荐流程

### 1. 先判断能否复用现有环境

先查看当前共享环境：

```bash
./tools/env-bootstrap/bin/env-bootstrap list
```

重点判断：

- Python 版本是否一致
- 依赖是否与现有环境大体兼容
- 是否引入新的重型原生运行时

### 2. 能复用时

- 把依赖补进对应模块组的 `environment.yml`
- 必要时更新对应 manifest 的 `EDITABLES`
- 执行一次 `sync`

例如 UAV 模块：

```bash
./tools/env-bootstrap/bin/env-bootstrap sync \
  ./tools/env-bootstrap/manifests/uav-modules.env.sh
```

### 3. 不能复用时

- 在 `._envs/` 规划一个新名字
- 在 `tools/env-bootstrap/manifests/` 新建一个 manifest
- 为新模块组准备自己的 `environment.yml`

推荐命名示例：

- `uav-ros-py310`
- `uav-training-py310`
- `vision-gpu-py311`

### 4. 迁移完成后固定当前平台

导出当前平台 lock：

```bash
./tools/env-bootstrap/bin/env-bootstrap freeze-explicit \
  ./tools/env-bootstrap/manifests/uav-modules.env.sh \
  ./modules/uav/conda-linux-aarch64.lock
```

## 迁移到新机器时

建议顺序：

1. 拉取仓库
2. 运行 `env-bootstrap doctor`
3. 运行 `env-bootstrap sync`
4. 用 `env-bootstrap run` 验证关键导入

## 清理原则

- 历史遗留的子项目内 `.conda-env` 应在迁移完成后显式清理。
- `._envs/` 下的孤儿环境使用 `env-bootstrap prune` 管理。
- 不要手工删除仍被 manifest 引用的共享环境。

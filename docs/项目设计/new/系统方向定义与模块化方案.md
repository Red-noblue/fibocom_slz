# 系统方向定义与模块化方案

## 1. 先把系统边界说死

这个项目如果边界不先钉死，后面一定越做越歪，最后什么都想做，什么都做不成。

本系统明确**不负责**：

1. 飞控控制
2. 姿态控制
3. 电机/舵面闭环
4. 机载实时控制指令下发
5. 避障控制执行

本系统明确**负责**：

1. 基于天气、动力参数、任务参数进行任务级预测
2. 基于预测结果进行任务级优化
3. 输出可解释的预测结果、优化结果和风险结果
4. 提供可视化展示、仿真配置与结果对比界面

一句话说清楚：

```text
我们不是飞控系统，我们是天气驱动的无人机任务预测与优化系统。
```

---

## 2. 项目适用场景

当前场景不应再按“空旷环境下的理想路线”理解，而应明确为：

```text
城市高楼低空环境
```

这个场景意味着：

1. 高度是重要变量，不再只是一个固定数字
2. 建筑物会影响风场分布，尤其是低空绕流、风廊、遮挡效应
3. 不同高度的天气条件可能明显不同
4. 路线不仅有平面路径问题，还有高度层选择问题
5. 展示层不能只停留在 2D 鸟瞰图，必须支持 3D 表达

所以，后续所有设计都要默认这个前提：

```text
项目不是在“平面地图上飞一条线”，而是在城市三维空间里做任务预测与优化。
```

---

## 3. 系统本质：预测层 + 优化层

这个系统本质上只有两层，不要乱加概念。

### 3.1 预测层

输入：

- 无人机动力参数
- 电池参数
- 载荷参数
- 起点终点
- 起飞时间
- 路线约束
- 高度约束
- 天气数据

输出：

- 总能耗预测
- 单位距离能耗预测
- 可达航程
- 可达飞行时间
- 风险提示
- 分段时序结果

### 3.2 优化层

在预测层之上，搜索更优解。

输出不再只是“能飞不能飞”，而是：

- 哪条路线更优
- 哪组高度更优
- 哪条速度变化曲线更优
- 哪组参数配置更优

所以后续所有“最佳方案”都应理解为：

```text
优化层调用预测层，对候选方案做评估后筛出更优解。
```

## 4. 推荐的系统模式

这部分是整个系统最重要的“业务模式定义”。

前端输入页、仿真配置页、结果展示页，都应该围绕这些模式组织。

### 模式一：基线评估模式

定义：

- 固定路线
- 固定航速
- 固定高度策略
- 固定其他参数

输出：

- 该任务在当前天气下的标准预测结果

作用：

- 作为所有对比和优化的基准方案

### 模式二：3D 路线优化模式

定义：

- 固定起点终点
- 固定或约束速度范围
- 搜索最佳三维路线

输出：

- 基线路线
- 优化路线
- 两者能耗差异
- 两者风险差异

说明：

这里的“3D 路线”不是简单平面绕行，而是：

- 平面路径变化
- 不同高度层切换
- 在城市空间中的通行走廊选择

### 模式三：速度曲线优化模式

定义：

- 固定路线
- 固定起点终点
- 固定其他条件
- 优化沿途速度变化曲线

输出：

- 固定速度方案
- 分段变速方案
- 各分段速度配置
- 总能耗/总风险对比

说明：

这非常适合你们当前的系统定位，因为它不碰飞控，只做任务级速度策略优化。

### 模式四：参数搜索/消融模式

定义：

- 固定大部分参数
- 只改变某一组参数
- 对不同配置做对比

可变参数包括：

- 高度配置
- 起飞时间
- 巡航速度
- 载荷
- 电池容量
- 路线约束

输出：

- 最优参数组合
- 参数敏感性排序
- 消融实验对比图

说明：

这个模式非常关键，因为它天然适合论文、答辩、技术报告和系统解释。

## 5. 输入端系统该怎么定义

你们真正需要的不是一个“地图输入框”，而是一个：

```text
仿真场景配置系统
```

也可以叫：

```text
任务配置与仿真输入端
```

### 输入端系统最少要有五块

#### 5.1 任务配置区

- 起点
- 终点
- 起飞时间
- 任务模式
- 是否固定路线
- 是否固定航速

#### 5.2 飞行器配置区

- 机型
- 电池容量
- 载荷重量
- 巡航速度范围
- 高度范围
- 动力学相关参数

#### 5.3 天气配置区

- 天气数据源
- 时间窗口
- 是否启用多高度天气
- 是否启用地面观测融合

#### 5.4 优化目标区

- 最低能耗
- 最低风险
- 最大剩余电量
- 最大可达性

#### 5.5 约束条件区

- 禁飞区
- 最大风速阈值
- 最大高度
- 最小高度
- 固定路线/可变路线
- 固定速度/可变速度

---

## 6. 输出端系统该怎么定义

输出端不能只给一个“最佳值”，否则非常廉价，也解释不清。

输出端应当是一个：

```text
结果分析与对比系统
```

### 输出端最少要有四类结果

#### 6.1 单次预测结果

- 总能耗
- 单位距离能耗
- 可达航程
- 可达时间
- 任务是否可完成
- 风险告警

#### 6.2 方案对比结果

- 基线方案 vs 优化方案
- 固定速度 vs 变速曲线
- 固定高度 vs 高度优化
- 固定参数 vs 参数搜索结果

#### 6.3 解释结果

- 主要影响因子
- 哪些天气条件导致能耗上升
- 哪些分段最危险
- 哪些参数最敏感

#### 6.4 消融实验结果

- 去掉高度信息的结果
- 去掉多高度天气后的结果
- 去掉风场特征后的结果
- 固定单一变量后的结果

---

## 7. 为什么前端必须支持 3D

如果系统后续拓展到高度信息，那么前端就不能继续假装世界是平的。

原因很简单：

1. 起点终点是在城市空间中，不只是地图上的两个点
2. 路径可能会换高度层
3. 高楼环境里的风场和风险区本来就是三维分布
4. 速度、风险、天气、路线的关系在 2D 图里会被压扁

所以前端必须具备：

- 2D 结果分析能力
- 3D 空间展示能力

不是二选一，而是两者同时存在。

---

## 8. 前端展示应该采用什么技术

这里不要拍脑袋，直接按问题类型选技术。

### 8.1 主 3D 地理引擎：CesiumJS

原因：

1. 你们的问题核心是**地理空间三维可视化**
2. 需要经纬度、高度、路线、城市空间位置表达
3. 后续可能要叠加风场层、风险层、高度层、航迹层

CesiumJS 最适合做：

- 城市三维地理场景
- 3D 航线展示
- 高度剖面表达
- 路线回放
- 多图层叠加

### 8.2 常规结果页：普通 Web 页面

不要犯蠢把所有页面都做成重 3D。

普通结果页继续用常规前端页面承接：

- 参数配置
- 表格结果
- 消融实验结果
- 对比分析
- 风险说明

这部分应该保持轻、快、可读。

### 8.3 图表层：普通 2D 图表库

用于：

- 电池余量曲线
- 风速曲线
- 高度曲线
- 速度曲线
- 能耗对比图
- 参数敏感性图

这里不需要上 3D，2D 图最清楚。

## 9. 前端系统推荐结构

不要再把页面理解成“一个地图页”。

推荐前端结构如下：

### 页面一：任务与仿真配置页

作用：

- 选择任务模式
- 配置输入参数
- 选择天气条件
- 选择优化目标

### 页面二：预测结果总览页

作用：

- 展示单次预测结果
- 展示方案是否可达
- 展示关键风险提示

### 页面三：3D 路线与城市空间展示页

作用：

- 展示路线
- 展示高度变化
- 展示城市空间位置
- 展示风险区/天气层

### 页面四：曲线分析页

作用：

- 能耗曲线
- 电池余量曲线
- 风速变化曲线
- 高度变化曲线
- 速度配置曲线

### 页面五：消融实验与参数对比页

作用：

- 展示不同方案差异
- 展示不同参数组合效果
- 展示解释性分析

## 10. 你们现在应该采用什么实现策略

你提得对，不要按“做完第一阶段就扔掉”的思路搞。

正确策略不是阶段性 demo 思路，而是：

```text
模块化实现，逐步接入能力，但前期模块不作废。
```

所以实现原则应该是：

### 10.1 先确定稳定接口

最先稳定的是：

- 输入任务对象
- 天气输入对象
- 预测输出对象
- 优化输出对象

### 10.2 预测层和展示层彻底解耦

预测层只负责：

- 算
- 输出结构化结果

展示层只负责：

- 读取结果
- 可视化
- 回放

### 10.3 3D 展示从第一版结构里就留接口

哪怕第一天只显示基础场景，也必须在架构上预留：

- 3D 路线对象
- 高度层对象
- 天气层对象
- 风险层对象

否则后面一定推翻重来。

## 11. 当前最合理的方向结论

可以直接定为下面这句话：

```text
本项目是面向城市高楼低空环境的无人机任务级天气驱动预测与优化系统。
系统与飞控解耦，输入任务参数、动力参数、天气参数和约束条件，
输出预测结果、优化结果、风险结果与可视化结果。
```

进一步压缩成最核心的三句话：

1. 我们不做飞控，我们做任务级预测与优化。
2. 我们的输入端是仿真场景配置系统，不是单一地图页。
3. 我们的展示端必须同时支持 2D 分析和 3D 城市空间表达。

# 仿真联合运行总体设计

## 1. 文档目的

本文档用于明确以下问题：

1. 为什么要采用“服务器侧高保真仿真 + fibo 开发板侧业务系统”的联合运行方式
2. 服务器侧与开发板侧各自承担什么职责
3. 两侧之间需要什么通信协议和数据流
4. 在当前服务器资源暂时不足的情况下，后续真正落地时应如何推进

本文档不是飞控设计文档，也不是单个仿真器安装文档，而是项目级联合运行设计文档。

---

## 2. 项目边界

本项目当前边界必须明确，不然最后一定做成四不像。

### 2.1 本项目不负责

1. 实际无人机飞控闭环控制
2. 姿态控制与电机控制
3. 避障执行控制
4. 真实无人机硬件接入主链路
5. 将优化结果直接转换为底层控制指令

### 2.2 本项目负责

1. 基于天气、任务参数、动力参数进行任务级预测
2. 基于预测结果进行任务级优化
3. 对预测/优化结果做仿真验证
4. 对运行状态、仿真状态和结果进行可视化展示
5. 形成可对外讲清楚的“高保真数字孪生验证链路”

一句话总结：

```text
本项目不是飞控系统，而是天气驱动的无人机任务预测、优化与仿真验证系统。
```

---

## 3. 为什么要采用“服务器 + 开发板”联合运行

这个决定不是锦上添花，而是现实约束下最合理的工程方案。

### 3.1 原因一：高保真仿真吃算力

如果后续使用：

- PX4 + Gazebo
- Isaac Sim
- 城市高楼三维环境
- 多高度天气场
- 路线/速度/参数优化

那么嵌入式开发板本机算力大概率扛不住。

尤其是 Isaac Sim，本来就更适合跑在独立工作站或服务器上。

### 3.2 原因二：开发板有自身价值，不应被重仿真拖死

fibo 开发板更适合承担：

1. 任务配置入口
2. Web 前端与结果展示
3. 预测/优化业务服务
4. 运行状态监控
5. 结果缓存与本地展示
6. 和未来硬件展示链路衔接

也就是说，开发板应该当：

```text
边缘业务控制台
```

而不是当：

```text
高保真仿真主机
```

### 3.3 原因三：即使没有真实无人机，也能补足可信度

当前项目没有真实无人机不是致命问题，只要补上一条可靠的叙述：

```text
我们当前不连接真实无人机，而是连接远端高保真数字孪生仿真环境。
```

这样系统仍然能完成：

- 任务配置
- 预测优化
- 仿真验证
- 状态回传
- 可视化展示

评委看到的就不是“空网页”，而是一套“边缘端 + 云端仿真端”的联合系统。

## 4. 总体架构

### 4.1 总体结构

```text
fibo 开发板
  ├─ 任务配置与仿真输入系统
  ├─ 天气驱动预测与优化系统
  ├─ 结果分析与网站展示系统
  ├─ 联合运行控制台
  └─ 通信网关

服务器
  ├─ 仿真编排服务
  ├─ PX4 + Gazebo 仿真底座
  ├─ Isaac Sim 高保真三维场景
  ├─ 天气场/环境场模块
  ├─ 日志与产物记录模块
  └─ 结果导出模块
```

### 4.2 两侧角色定义

#### 开发板侧角色

开发板侧是：

```text
任务主控 + 展示主控 + 业务主控
```

主要负责：

1. 用户输入任务参数
2. 选择仿真模式
3. 发起仿真任务
4. 接收服务器状态
5. 渲染结果页面
6. 展示回放与风险

#### 服务器侧角色

服务器侧是：

```text
仿真执行主控
```

主要负责：

1. 启动/停止仿真
2. 运行 PX4/Gazebo
3. 运行 Isaac Sim
4. 生成天气场和环境场
5. 汇总状态
6. 导出仿真结果产物

---

## 5. 为什么不是“开发板直接连接 PX4/Gazebo/Isaac Sim”

如果让开发板直接分别连接多个仿真模块，整个系统很快会变乱：

1. 状态来源不统一
2. 错误处理分散
3. 时钟同步难做
4. 前端无法知道哪个模块挂了
5. 接口膨胀得很快

所以中间必须有一层统一入口：

```text
仿真编排服务（Simulation Orchestrator）
```

开发板只和这个服务通信。

仿真编排服务统一对接：

- PX4/Gazebo
- Isaac Sim
- 天气场模块
- 日志记录
- 产物导出

这个设计非常关键，不然系统后续一定失控。

## 6. 联合运行的核心逻辑

### 6.1 标准运行流程

#### 第一步：用户在开发板侧配置任务

用户输入：

- 起点
- 终点
- 起飞时间
- 电池参数
- 载荷参数
- 高度范围
- 速度范围
- 天气源配置
- 仿真模式
- 优化目标

#### 第二步：开发板侧生成任务对象

形成标准任务结构：

```text
任务参数 + 飞行器参数 + 天气参数 + 约束条件 + 优化目标
```

#### 第三步：开发板侧向服务器发起仿真任务

服务器收到任务后，由仿真编排服务：

1. 分配任务 ID
2. 准备天气场
3. 准备 PX4/Gazebo
4. 准备 Isaac Sim 场景
5. 启动状态采集

#### 第四步：服务器侧开始仿真

这一阶段会持续向开发板回传：

- 当前状态
- 当前阶段
- 当前位姿
- 当前电池
- 当前天气
- 当前风险
- 当前异常情况

#### 第五步：开发板侧实时展示

开发板前端显示：

- 仿真是否启动成功
- 当前运行到哪一步
- 无人机状态
- 当前天气信息
- 当前风险等级

#### 第六步：服务器侧导出结果产物

仿真结束后，服务器导出：

- `summary.json`
- `timeseries.csv`
- `map.html`
- 其他日志/截图/录像

#### 第七步：开发板侧读取结果并展示

开发板侧将这些产物作为展示站点输入：

- 总览页
- 地图页
- 回放页
- 风险页
- 对比页

## 7. 通信设计

这个部分一定要定，不然“联合运行”只是一句空话。

### 7.1 必须存在通信协议

服务器和开发板之间必须有协议，不然开发板无法知道：

1. 仿真是否启动
2. 当前执行到哪一步
3. 是否异常退出
4. 当前天气和飞行状态是什么
5. 仿真结果何时产生

所以答案是：

```text
必须设计通信协议，而且应该尽早设计。
```

### 7.2 推荐分两层协议

#### 控制层协议

用途：

- 创建任务
- 启动仿真
- 暂停仿真
- 停止仿真
- 查询状态
- 拉取结果

推荐技术：

```text
HTTP / REST
```

原因：

1. 简单
2. 易调试
3. 易做前端
4. 当前阶段足够

#### 实时状态层协议

用途：

- 推送仿真进度
- 推送飞行状态
- 推送实时风险
- 推送异常信息

推荐技术：

```text
WebSocket
```

原因：

1. 前端实时展示方便
2. 状态推送自然
3. 不需要前端轮询浪费资源

### 7.3 仿真内部通信

服务器内部不建议都自己乱写协议。

建议：

```text
PX4 / Gazebo / 相关节点之间优先走 ROS 2
```

即：

- 服务器外部接口：REST + WebSocket
- 服务器内部仿真总线：ROS 2

这才是合理分层。

## 8. 最小协议草案

### 8.1 创建仿真任务

开发板发给服务器：

```json
{
  "task_id": "sim_20260512_001",
  "mode": "baseline",
  "vehicle": {
    "battery_wh": 130,
    "payload_g": 250,
    "speed_mps": 8.0,
    "altitude_m": 60
  },
  "mission": {
    "start": [39.9598, 116.3520],
    "end": [40.15608, 116.28333],
    "departure_time": "2026-05-12T15:30:00"

  "constraints": {
    "fixed_route": true,
    "fixed_speed": true
  },
  "weather": {
    "source": "forecast",
```

```json

  "status": "queued"

```

## 9. 数据接口与产物接口

联合运行如果没有稳定产物接口，也一样会烂。

### 9.1 对展示层最重要的标准产物

建议沿用当前已形成的产物结构：

1. `summary.json`
2. `timeseries.csv`

可选补充：

3. `map.html`
4. `site/`
5. `replay.mp4`
6. `logs/`

### 9.2 为什么这两个产物最重要

因为它们刚好能让：

- 预测系统输出结构化结果
- 展示系统稳定消费结果

这样服务器侧仿真跑完后，开发板侧就不需要理解 PX4/Gazebo/Isaac Sim 细节，只需要吃结果。

这个分层非常干净。

## 10. 服务器侧的内部模块建议

服务器侧建议至少拆成下面五块。

### 10.1 仿真编排服务

职责：

- 接收任务
- 调度子模块
- 管理任务生命周期
- 汇总状态

### 10.2 飞行仿真底座

职责：

- 运行 PX4 + Gazebo
- 提供主飞行仿真真值

说明：

这里应当作为主仿真真值来源。

### 10.3 高保真三维场景模块

职责：

- 运行 Isaac Sim
- 展示城市高楼空间场景
- 承担高保真视觉/数字孪生表达

说明：

这里是高保真场景层，不建议作为唯一主飞行真值。

### 10.4 天气场/环境场模块

职责：

- 生成多高度天气信息
- 生成三维风场
- 对接预测系统
- 对接仿真世界

说明：

这个模块很关键，因为：

```text
没有统一天气真值，就没有可信联合仿真。
```

### 10.5 结果记录与导出模块

职责：

- 保存过程日志
- 保存轨迹时序
- 保存最终结果
- 产出标准文件

---

## 11. 开发板侧的内部模块建议

开发板侧建议至少拆成下面五块。

### 11.1 输入端系统

职责：

- 用户配置任务
- 用户选择模式
- 用户配置天气与约束

### 11.2 预测与优化服务

职责：

- 生成基线预测
- 生成优化请求
- 对服务器结果做本地分析

### 11.3 联合运行控制台

职责：

- 发起任务
- 监控状态
- 控制暂停/停止
- 读取结果

### 11.4 展示系统

职责：

- 展示 GIS 页面
- 展示回放
- 展示任务状态
- 展示风险提示

### 11.5 通信网关

职责：

- 对接服务器 REST 接口
- 对接 WebSocket 状态流
- 做本地缓存与重连

## 12. 这样做的答辩价值

这条联合运行架构非常适合比赛和项目答辩。

因为你们可以很完整地说：

1. 我们当前没有真实无人机实物接入
2. 但我们没有停留在概念预测
3. 我们把系统接到了远端高保真仿真环境
4. 服务器侧负责数字孪生仿真
5. 开发板侧负责任务配置、预测优化和结果展示
6. 两侧通过统一协议形成完整闭环

这套表述比“我们就做了个网页和算法”强太多。

## 13. 当前阶段最现实的落地顺序

现在服务器暂时没有空闲资源，所以不要嘴上过瘾，手上乱做。

当前最现实的推进顺序应该是：

### 第一步：先把协议和数据结构定下来

优先确定：

1. 任务输入结构
2. 实时状态结构
3. 结果产物结构

### 第二步：开发板侧先做空跑链路

即使没有服务器，也先把：

- 输入端
- 状态页
- 结果页
- 协议客户端

这些都搭好。

此时服务器接口可以先用 mock 数据替代。

### 第三步：服务器资源空出来后再接真实仿真

真正接入：

- PX4/Gazebo
- Isaac Sim
- 天气场模块

### 第四步：再做联合回放和网站发布

把开发板前端正式变成：

```text
边缘控制台 + 数字孪生展示门户
```

---

## 14. 当前最重要的组织原则

整个联合运行设计，最重要的不是“把软件装起来”，而是守住以下三条原则。

### 原则一：开发板不碰重仿真

开发板只承担：

- 配置
- 控制
- 分析
- 展示

### 原则二：服务器对外只有一个统一入口

不要让开发板直接分别访问：

- PX4
- Gazebo
- Isaac Sim

开发板只连：

```text
仿真编排服务
```

### 原则三：结果产物标准化

必须统一：

- `summary.json`
- `timeseries.csv`

这样展示层、业务层、仿真层就不会互相污染。

## 15. 最终结论

当前推荐的正式表述如下：

```text
本项目采用“服务器侧高保真仿真 + fibo 开发板侧业务系统”的联合运行架构。
服务器侧负责 PX4/Gazebo、Isaac Sim、天气场和仿真执行；
开发板侧负责任务配置、预测优化、状态监控与网站展示。
两侧通过 REST + WebSocket 协议协同，形成一套不依赖真实无人机实物、
但能够借助远端高保真数字孪生环境完成验证的任务级预测与优化系统。
```

这句话后续可以直接拿去改写成：

1. 答辩口径
2. 系统架构页
3. 任务流程页
4. 通信接口设计页

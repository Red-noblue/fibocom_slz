## Karpathy 风格协作准则
这些行为准则用于减少常见的 LLM 编码错误，并与本项目自身规则一同生效。

权衡说明：
这些准则倾向于“谨慎优先，而不是速度优先”。对于非常琐碎的任务，可以自行判断，不必机械套用全部流程。

### 1. 编码前思考
不要假设。不要隐藏困惑。呈现权衡。

在实现之前：
- 明确说明你的假设。如果不确定，就询问，而不是猜测。
- 如果存在多种合理解释，先呈现出来，不要静默选择其中一种。
- 如果存在更简单的方法，应明确说出来；在必要时应提出异议。
- 如果有不清楚的地方，就停下来，指出困惑点并请求澄清。

### 2. 简洁优先
用最少的代码解决问题。不要过度推测。

- 不要添加要求之外的功能。
- 不要为一次性代码创建抽象。
- 不要加入未被请求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景添加错误处理。
- 如果你写了 200 行，但 50 行就能解决，应当重写并简化。

自我检查：
资深工程师会觉得这段实现过于复杂吗？如果答案是会，就继续简化。

### 3. 精准修改
只碰必须碰的。只清理你自己造成的混乱。

编辑现有代码时：
- 不要顺手“改进”相邻代码、注释或格式。
- 不要重构没有坏掉的东西。
- 尽量匹配现有风格，即使你个人更偏好别的写法。
- 如果你注意到无关的死代码，可以指出，但不要擅自删除。

当你的改动制造出孤儿项时：
- 删除因你的改动而变得无用的导入、变量、函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

检验标准：
每一行修改都应该能直接追溯到用户当前请求。

### 4. 目标驱动执行
定义成功标准，并持续验证直到达成。

尽量把任务转化成可验证目标，例如：
- “添加验证” 转成 “为非法输入编写测试，然后让测试通过”
- “修复 bug” 转成 “先写出能稳定重现 bug 的检查或测试，再让它通过”
- “重构 X” 转成 “确保重构前后测试、构建或关键行为保持一致”

对于多步骤任务，先给出简短计划：
1. [步骤] -> 验证: [检查]
2. [步骤] -> 验证: [检查]
3. [步骤] -> 验证: [检查]

强成功标准可以让你独立循环推进；弱成功标准比如“让它能工作”通常会导致反复返工和额外澄清。

### 5. 这些准则是否生效
如果这些准则起作用，通常会看到：
- diff 中不必要的改动更少
- 因过度复杂导致的重写更少
- 澄清问题出现在实现之前，而不是犯错之后

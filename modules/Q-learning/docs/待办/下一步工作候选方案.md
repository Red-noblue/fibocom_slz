# 下一步工作候选方案

本文记录 `modules/Q-learning` 后续工作的候选路线。当前已明确：本目录短期服务 Q-learning 课程作业，中期定位为低空无人机端边云协同中的策略学习 / 计算卸载决策层。

## 推荐执行顺序

建议按以下顺序推进：

```text
方案 1 -> 方案 2 -> 方案 3 或方案 5 -> 方案 4
```

原因是先把当前基础实验做稳，再增加低空任务语义；之后根据阶段目标选择更强动作空间、更多算法对比，最后再做跨模块接口契约。

## 方案 1：强化当前 Q-learning 基线实验

目标是在不接入外部模块的前提下，把当前 `Q-learning` 目录内的实验做扎实。

主要内容：

- 批量运行 `balanced`、`good_coverage`、`weak_coverage`、`intermittent_coverage`、`congested_edge` 场景。
- 输出每个场景下的策略对比结果。
- 汇总 Q-learning、Local Only、Offload Only、Rule-Based 的平均 reward、平均时延、平均能耗、队列长度、处理任务数和卸载比例。
- 保留每个场景的 `decision_trace.csv`、`learning_curve.csv`、`policy_table.csv`，方便解释策略行为。

成功标准：

```text
一条命令生成多场景结果，能清楚回答“什么时候本地算、什么时候卸载”。
```

## 方案 2：加入低空任务语义

目标是让状态和 reward 更像真实低空经济任务。

主要内容：

- 在状态中加入 `task_urgency`、`data_sensitivity` 或 `area_risk`。
- 在 reward 中加入超时惩罚、安全风险惩罚、敏感数据上传惩罚。
- 让策略能区分避障告警、巡检识别、日志上传等不同任务类型。

成功标准：

```text
策略能表现出紧急任务倾向低时延处理、敏感数据减少上传、高风险区域更保守的行为。
```

## 方案 3：扩展为端-边-云三层卸载

目标是从课程级本地 / 边缘卸载升级到更贴近通感算一体化的端边云协同。

主要内容：

- 将动作扩展为本地、边缘、云端、混合计算。
- 建模云端高算力但高时延，边缘中等时延，本地低时延但算力有限。
- 对比不同链路质量、边缘负载和任务紧急度下的动作选择。

成功标准：

```text
策略能在不同状态下区分本地、边缘和云端，而不是只做二选一。
```

## 方案 4：设计与其他模块的离线接口契约

目标是暂不修改其他模块，但为后续接入做好输入输出格式。

主要内容：

- 设计 `network_trace.csv`，用于承接无线电地图或网络覆盖模块输出。
- 设计 `energy_trace.csv`，用于承接天气能耗模块输出。
- 设计 `task_trace.csv`，用于描述任务队列、任务类型、紧急度、数据敏感性。
- 让策略模块可以从 CSV trace 读取状态序列，而不是只依赖内部随机采样。

成功标准：

```text
Q-learning 策略模块可以消费外部 trace 文件，具备后续接入其他模块的稳定边界。
```

## 方案 5：算法升级对照

目标是证明本目录不局限于 Q-learning，而是策略学习实验工作区。

主要内容：

- 保留当前表格型 Q-learning 作为可解释基线。
- 增加规则策略、上下文 bandit、DQN 中的一种作为对照。
- 比较低算力、低复杂度方法与高效果方法之间的取舍。

成功标准：

```text
同一环境下输出多算法对比，说明低算力 / 中算力 / 高效果算法各自适用场景。
```

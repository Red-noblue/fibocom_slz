# 成果总结与推荐路线

本文总结 2026-07-03 当前设备上 radio-map 模型部署的阶段性成果。统计范围只包含真实 radio-map 模型或由其转换得到的 DLC / QNN `.so` 产物。

## 当前结论

当前已经形成三条有价值的部署路线：

| 优先级 | 路线 | 状态 | 说明 |
| --- | --- | --- | --- |
| 默认主线 | SNPE GPU INT8 DLC | 推荐使用 | 精度损失小，速度稳定，部署和复现成本低。 |
| 稳定备选 | SNPE GPU FP32 DLC | 可用 | 精度基线最好，速度略慢于 INT8 GPU。 |
| 速度副主线 | SNPE DSP INT8 DLC | 可选 | 速度约为 GPU 的 `3.6-3.7x`，但输出偏差更大。 |
| 备用验证线 | QNN GPU `.so` | 可用但非主线 | 可运行完整 radio-map QNN `.so`，但速度慢于 SNPE GPU INT8。 |
| 研究线 | QNN DSP Pad+INT8 `.so` | 改写版可执行 | 10 样本平均约 `90 ms`，但精度弱于 SNPE DSP INT8；未改写 `.so` 仍失败在 Pad / op validation。 |

## 不计入可用加速链路的情况

| 链路 | 当前状态 | 原因 |
| --- | --- | --- |
| SNPE HTP | 疑似 CPU fallback | 能初始化，但耗时接近 CPU，缺少硬件活动证据。 |
| SNPE NPU | 疑似 CPU fallback | 能初始化，但耗时接近 CPU。 |
| SNPE DSP FP32 | 不作为 DSP 加速路线 | 日志提示 `SocModel doesn't support FP16`，耗时接近 CPU。 |
| QNN DSP 未改写完整 `.so` | 不是 fallback，而是失败 | 加载 `libQnnHtp.so` 后失败在 `_enc1_c1__prepad_Pad` / `MODEL_GRAPH_OP_VALIDATION_ERROR`。 |
| QNN DSP Pad+INT8 改写 `.so` | 可执行但非主线 | 完整模型已能运行，10 样本 RMSE `4.571 dB`，弱于 SNPE DSP INT8。 |
| QNN HTP / NPU runtime 字符串 | SDK 不支持 | 日志提示 `runtime_type is not supported`。 |
| QNN + DLC | SDK 不支持 | 日志提示 `QNN not support using modeldlc now`。QNN 路线必须使用 `.so`。 |

## 为什么把 SNPE GPU INT8 作为默认主线

SNPE GPU INT8 在 20 个真实 test 样本上相对 FP32 GPU 的平均输出 RMSE 只有 `0.184 dB`，而整体标签 RMSE 从 `3.279 dB` 增加到 `3.299 dB`。这个变化小于模型自身误差量级，且部署路径简单、返回结果稳定。

## 为什么 SNPE DSP INT8 是速度副主线

SNPE DSP INT8 的平均耗时约 `83 ms`，相对 INT8 GPU 约 `3.6-3.7x` 加速。`strace` 已看到 `/dev/adsprpc-smd-secure`、`libSnpeHtpV68Stub.so`、`libSnpeHtpV68Skel.so` 和 FastRPC ioctl，因此不能再按普通 CPU fallback 处理。

但 DSP 输出偏差明显高于 GPU INT8：20 样本相对 FP32 GPU 的平均 RMSE 为 `1.633 dB`，100 样本相对 INT8 GPU 的平均 RMSE 为 `1.715 dB`。当前认为该误差可以接受，因此 DSP INT8 可以作为高性能模式；默认稳妥精度模式仍选择 GPU INT8。

## 推荐口径

```text
默认模式：SNPE GPU INT8
高性能模式：SNPE DSP INT8，接受约 1.7 dB RMSE 输出差异时使用
精度基线：SNPE GPU FP32
备用验证：QNN GPU .so
继续研究：QNN DSP Pad+INT8 .so / HTP-CDSP 改图精度优化
```

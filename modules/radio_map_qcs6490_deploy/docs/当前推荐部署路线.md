# 当前推荐部署路线

本文档给出当前 SC171V3 / QCS6490 设备上部署无线电地图估计模型的执行口径。目标是服务本机部署、运行测试和后续专项推进，不承担训练任务。

## 一句话结论

技术路线探索阶段已关闭，当前进入“复现实验固化 + 指标补强 + 可选后端专项”阶段。50/100 样本 INT8 GPU/DSP 批量补强实验已完成；这些实验用于增强可信度，不再作为路线成立的前置条件。

当前正式部署路线是：

```text
默认主线：真实 M2 特征样本 -> radio_map_liteunet_quantized.dlc -> Fibo AI Stack Python SDK -> SNPE GPU -> Adreno GPU INT8 推理
速度副主线：真实 M2 特征样本 -> radio_map_liteunet_quantized.dlc -> Fibo AI Stack Python SDK -> SNPE DSP -> Hexagon/CDSP INT8 推理
备选验证：真实 M2 特征样本 -> libradio_map_liteunet_qnn.so -> Fibo AI Stack Python SDK -> QNN GPU
专项研究：真实 M2 特征样本 -> Pad+INT8 改写 QNN .so -> Fibo AI Stack Python SDK -> QNN DSP -> HTP/CDSP
```

当前不把 SNPE HTP/NPU 或 QNN DSP 作为主线前置条件。SNPE DSP INT8 已具备速度副主线条件，但精度相对 GPU INT8 更差；QNN DSP Pad+INT8 改写版已经能执行完整模型，不过精度弱于 SNPE DSP INT8，继续归入 HTP/CDSP 专项。

0703 成果总览、指标和复现指令见：

```text
docs/0703-完成DSP、SNPE等部署/README.md
docs/各路线运行示例.md
```

## 正式主线

使用系统 Python 3.8 和本机 Fibo AI Stack SDK：

```bash
python3 modules/radio_map_qcs6490_deploy/scripts/run_snpe_gpu_inference.py \
  --sample-id 166_k0 \
  --runtime GPU \
  --warmup 1 \
  --repeat 2
```

批量复现实验：

```bash
python3 modules/radio_map_qcs6490_deploy/scripts/batch_snpe_int8_repro.py \
  --limit 50 \
  --warmup 1 \
  --repeat 1 \
  --output-dir modules/radio_map_qcs6490_deploy/outputs/int8_repro_test50_$(date +%Y%m%d_%H%M%S)
```

## 部署就绪检查

先运行只读检查：

```bash
python3 modules/radio_map_qcs6490_deploy/scripts/check_deploy_readiness.py
```

该脚本检查：

- 系统 Python 是否为 Fibo SDK 兼容的 CPython 3.8。
- `fiboaisdk`、`InferAPI`、`InferParams` 是否可导入。
- DLC、ONNX、M2 特征样本、`samples.csv`、`stats.json` 是否存在。
- KGSL / Adreno GPU 设备节点是否存在。
- SNPE GPU/DSP/NPU fallback 判据 JSON 是否符合当前结论。
- 100 样本 SNPE GPU 基准是否存在且 `100/100` 成功。
- x86_64 QEMU binfmt 是否已指向 QEMU 8.2.2。

当前如果只做 SNPE GPU 推理，QEMU 不是运行前置条件；如果继续 HTP/CDSP 量化专项，必须先处理 QEMU。

## CPU Fallback 判据

不要只看 `Init=0` 或 `Execute=0`。当前硬性判据脚本是：

```bash
python3 modules/radio_map_qcs6490_deploy/scripts/judge_backend_fallback.py \
  --run-dir modules/radio_map_qcs6490_deploy/outputs/snpe_backend_judgement_20260703_044730 \
  --runtime-key GPU
```

当前判定：

| runtime | verdict | 结论 |
| --- | --- | --- |
| SNPE GPU | `hardware_backend_confirmed` | 真实 GPU 后端成立。 |
| SNPE DSP FP32 | `backend_loaded_no_acceleration_evidence` | 加载过后端，但 FP32 路线无有效加速证据。 |
| SNPE DSP INT8 | `hardware_backend_confirmed` | `strace` 和耗时均支持真实 CDSP/HTP 加速。 |
| SNPE HTP / NPU | `suspected_cpu_fallback` | 初始化成功不等于真实加速，仍疑似 CPU fallback。 |

SNPE GPU 的关键证据是：日志有 GPU runtime、相对 CPU 约 `3.82x` 加速、`kgsl_gpubusy` 明确增长、输出与 CPU 只有很小混合精度差异。SNPE DSP INT8 的关键证据是：打开 `/dev/adsprpc-smd-secure`、`/dsp/cdsp`、`libSnpeHtpV68Stub.so`、`libSnpeHtpV68Skel.so`，并出现 FastRPC DMA 分配；长 repeat 平均耗时约 `80 ms`。

## QEMU 与量化专项

当前本机同时存在：

```text
/usr/bin/qemu-x86_64-static                  -> 4.2.1
/usr/local/bin/qemu-x86_64-static-8.2.2      -> 8.2.2
```

检查命令：

```bash
modules/fiboaistack_229_env/scripts/check_qemu_x86_64.sh
```

若标准 `qemu-x86_64` 仍指向旧 QEMU，执行：

```bash
modules/fiboaistack_229_env/scripts/register_qemu82_binfmt.sh
```

当前已验证：

- 最小 `plain_conv_int8` 在 QEMU 8.2.2 下可完成 `qnn-onnx-converter --input_list` 量化转换。
- 完整 radio-map 模型可通过 `snpe-dlc-quantize` 生成 SNPE INT8 DLC。
- 完整 radio-map 模型可通过 `qairt-quantizer` 生成 QAIRT INT8 DLC。
- 量化产物已完成 SNPE GPU / DSP 初始化和真实样本精度测试。

## 后续执行顺序

1. 将 SNPE GPU INT8 作为正式部署主线，保留 FP32 GPU 作为基线对照。
2. 将 `check_deploy_readiness.py` 作为每次部署前的入口检查。
3. 将 SNPE DSP INT8 作为速度副主线，使用时必须同时披露精度损失和 FastRPC/CDSP 证据。
4. QNN DSP 已完成 Pad+INT8 首轮图改写并跑通完整模型，但当前精度较弱；后续重点转向 padding 替代策略和精度损失定位，不再用未改写 full `.so` 作为可部署路线。
5. SNPE HTP/NPU 继续按疑似 CPU fallback 处理，除非后续取得独立耗时、日志和设备计数证据。

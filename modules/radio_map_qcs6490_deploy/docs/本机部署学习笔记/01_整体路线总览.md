# 整体路线总览

## 这篇解决什么问题

这篇用于解释当前复现工作的整体方法：为什么要先把论文模型变成 ONNX，再通过 Fibo AI Stack 转成 Qualcomm/Fibo 可运行格式，最后在当前 QCS6490 设备上用本机 SDK 做推理验证。

当前工作不是重新训练模型，而是把已经得到的模型部署到本设备上运行。因此重点从“训练框架兼容”转向“模型格式转换、板卡后端选择、设备侧推理验证”。

## 当前总流程

当前已经探索出的主链路如下：

```text
论文项目模型
  |
  |  导出
  v
ONNX: radio_map_liteunet.onnx
  |
  |  Fibo AI Stack 2.29 / SNPE 转换工具
  v
DLC: radio_map_liteunet.dlc
  |
  |  当前设备原生 fiboaisdk
  v
SNPE GPU / CPU / DSP / NPU 推理测试
```

另有一条 QNN 备用链路：

```text
ONNX: radio_map_liteunet.onnx
  |
  |  qnn-onnx-converter
  v
QNN C++/bin/json
  |
  |  本机 aarch64 编译
  v
QNN 模型库: libradio_map_liteunet_qnn.so
  |
  |  当前设备原生 fiboaisdk
  v
QNN CPU / GPU / DSP 推理测试
```

目前主推第一条 SNPE GPU 路线，因为它的速度、日志证据和实际可用性最好。

## 为什么不直接跑论文项目

论文项目 `modules/radio-map-estimation-workbench` 更接近训练、评估和研究代码。它关注的是 PyTorch 模型、数据集、训练过程、指标和实验复现。

当前设备部署关注的是另一件事：

- 模型能不能被 Fibo AI Stack 转换。
- 转换后的模型能不能被本机 SDK 加载。
- 本机 Adreno GPU、Hexagon/CDSP/HTP 等后端是否真的参与推理。
- 推理速度和输出是否合理。

因此我们没有把部署逻辑直接塞回论文项目，而是新建了：

```text
modules/radio_map_qcs6490_deploy
```

这个目录专门服务当前设备部署验证，避免把训练项目和板卡部署适配混在一起。

## 为什么需要 ONNX

ONNX 在这里是中间交换格式。论文模型来自 PyTorch，但 Fibo AI Stack / Qualcomm SNPE/QNN 工具链不能直接吃 PyTorch checkpoint。通常需要先导出为 ONNX，再由厂商工具转换成设备侧推理格式。

本项目当前的 ONNX 文件是：

```text
modules/fiboaistack_229_env/inputs/radio_map_liteunet.onnx
```

它是转换工具链的输入，而不是最终部署格式。

## 为什么需要 DLC

DLC 是 SNPE 使用的模型容器格式。当前本机 `fiboaisdk` 中的 SNPE 路线可以直接加载 DLC，并通过 `runtime` 参数选择 CPU/GPU/DSP/NPU。

当前已经生成的 DLC 是：

```text
modules/fiboaistack_229_env/outputs/radio_map_liteunet.dlc
```

配套模型信息文件是：

```text
modules/fiboaistack_229_env/outputs/radio_map_liteunet.info.txt
```

当前推理探针默认输入输出为：

```text
input  -> shape [1, 256, 256, 7]
output -> shape [1, 256, 256, 1]
```

## 为什么 Docker 只用于转换

当前 Docker 容器里的 Fibo AI Stack 2.29 工具链主要提供这些转换命令：

```text
snpe-onnx-to-dlc
snpe-dlc-info
qnn-onnx-converter
qnn-model-lib-generator
qairt-quantizer
snpe-dlc-quantize
```

这些工具用于把 ONNX 转成设备侧模型格式。转换完成后，真正的推理测试是在本机原生环境执行：

```text
/usr/local/lib/python3.8/dist-packages/fiboaisdk
```

这样做的原因是：我们要验证当前板卡的真实硬件后端，例如 Adreno GPU、CDSP、HTP，而这些设备节点和驱动能力属于宿主机，不属于转换容器。

## 当前已经确认的实测结论

当前最可靠的路线是：

```text
framework=snpe
runtime=GPU
mode=5
model=modules/fiboaistack_229_env/outputs/radio_map_liteunet.dlc
```

关键证据包括：

- 日志显示 `Set gpu_float32_16_hybrid runtime succeeded`。
- 日志显示节点创建为 `all_all_infer_snpe_gpu_1.0.0`。
- 推理速度显著快于 CPU/DSP/NPU。
- 输出均值与 CPU 有轻微差异，符合 GPU 混合精度路径特征。

QNN GPU 也可用，但当前速度慢于 SNPE GPU。QNN DSP 会加载 HTP/CDSP 相关库和配置；未改写 FP32 `.so` 会在 `Pad` 节点校验和 FP16/SoC 支持问题处失败。0703 后续的 Pad+INT8 改写版完整模型已经可以执行，但精度弱于 SNPE DSP INT8，因此仍作为专项研究线。

## 容易误解的点

### 返回码成功不等于硬件加速成功

例如 SNPE DSP 和 SNPE NPU 当前可以返回成功，但耗时接近 CPU，输出也和 CPU 完全一致，同时日志里有 `The SocModel doesn't support FP16` 等警告。因此不能仅凭 `Init=0` 或 `Execute=0` 认定它们真的用了 Hexagon/CDSP。

### Docker 不是最终运行环境

Docker 只是为了运行 Fibo AI Stack 2.29 的转换工具。真正的部署目标是当前设备本机原生环境。

### HTP/CDSP 不是简单改一个 runtime 字符串就能用

HTP/CDSP 对模型格式、量化、算子支持、后端配置和 SoC 能力都有要求。当前 QNN DSP 已经证明会尝试进入 HTP 后端；未改写模型不能完成加载，Pad+INT8 改写版可以完成推理但还需要解决精度损失。

## 当前工作边界

当前阶段建议先稳定 SNPE GPU 本机部署路线。除非明确要继续追求 CDSP/HTP，否则不应把主要时间花在 QNN DSP/INT8 量化问题上。

后续如果要继续优化，可以围绕三件事展开：

- 做真实样本输入输出对齐，而不只使用随机输入。
- 固化 SNPE GPU 推理脚本和评估入口。
- 单独研究量化和 HTP/CDSP 支持问题。

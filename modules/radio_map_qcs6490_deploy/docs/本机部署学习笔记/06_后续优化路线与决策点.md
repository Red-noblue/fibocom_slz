## 当前结论

当前最稳妥的本机部署主线是：

```text
真实 M2 特征样本 -> SNPE DLC -> Fibo AI Stack Python SDK -> runtime=GPU -> Adreno GPU 推理
```

已经验证：

- 本设备是 `YUPIKP-IOT`，SoC ID 为 `498`，GPU 为 `Adreno643v1`。
- 系统存在 `/dev/kgsl-3d0`，可用于 Adreno GPU 路线。
- 系统存在 `/dev/adsprpc-smd`、`/dev/adsprpc-smd-secure` 和 Fibo SDK 的 `hexagon-v68` 库，说明 CDSP/HTP 通道具备基础环境。
- 当前 DLC 是 FP32 模型，`Quantizer command: N/A`，不是 INT8/HTP 专用模型。
- SNPE GPU 在真实样本 `166_k0` 上可用，平均推理约 300 ms，明显快于 CPU 的约 1.1 s。
- SNPE CPU/GPU 在真实样本上的输出差异很小，标准化空间 RMSE 约 `0.00115`，还原到 dBm 后 CPU/GPU 差异 RMSE 约 `0.0147 dB`。
- SNPE GPU 已扩展到 20/50/100 个 test 样本批量复现实验，100 样本全部成功，平均推理约 `331 ms`，相对 CPU 平均加速约 `3.87x`。

## 关于 SNPE DSP/NPU 的判断

“SNPE DSP/NPU”不是硬件模块名称，而是 Fibo/SNPE 封装层暴露的运行时选择。

更准确的层级是：

```text
硬件/子系统: Hexagon / CDSP / HTP
系统通道: FastRPC, adsprpc, cdsprpcd
推理框架: SNPE, QNN
封装参数: runtime=CPU/GPU/DSP/NPU
```

当前设备确实具备 CDSP/HTP 相关环境，但当前不能说“SNPE DSP/NPU 已经成功跑在专用硬件上”。

### SNPE DSP

已有证据：

- 日志出现 `Set dsp_fixed8_tf runtime succeeded`。
- 同时出现 `Set cpu_float32 runtime succeeded`。
- 日志大量出现 `The SocModel doesn't support FP16`。
- 耗时接近 CPU。
- 输出和 CPU 完全一致或极接近。
- 当前 DLC 是 FP32，未量化。

判断：

- SNPE DSP 至少尝试设置 DSP runtime。
- 但当前模型没有表现出真实 DSP/HTP 加速。
- 不能凭 `Init=0` 或 `Execute=0` 认定已经用了 Hexagon/CDSP。

### SNPE NPU

已有证据：

- 当前实测主要看到 `Set cpu_float32 runtime succeeded`。
- 未看到类似 GPU 的明确硬件 runtime 证据。
- 耗时接近 CPU。
- 输出接近 CPU。

判断：

- 当前 SDK 中 `runtime=NPU` 更像一个逻辑字符串或兼容入口。
- 对当前模型和当前设备，不建议把 SNPE NPU 作为真实加速目标。

## 如何避免误判为 CPU fallback

判断一个后端是否真实可用，至少要同时看 4 类证据：

1. 日志是否出现明确后端，例如 `gpu_float32_16_hybrid`、`dsp_fixed8_tf`、`libQnnHtp.so`。
2. 耗时是否相对 CPU 明显下降。
3. 输出是否符合该后端精度特征，例如 GPU 混合精度相对 CPU 有极小差异。
4. 设备侧是否有对应节点活动，例如 KGSL 或 FastRPC/CDSP。

当前 SNPE GPU 满足前 3 点，且速度优势明显。SNPE DSP/NPU 不满足速度和输出特征要求。

当前 SDK 没有发现公开的“禁用 CPU fallback”开关。实际可执行策略是：

- 主线部署只使用已验证的 `framework=snpe`、`runtime=GPU`。
- 后端判断不只看返回码，必须同时看日志、耗时、输出差异和设备侧证据。
- 对 DSP/NPU，如果返回成功但耗时接近 CPU、输出完全等同 CPU，就按疑似 fallback 处理。
- 对 QNN DSP，如果日志出现 `libQnnHtp.so`、FastRPC/CDSP 通道和 `QnnBackend_validateOpConfig`，说明它不是简单 fallback，而是进入 HTP 后模型校验失败。
- 真正想让 DSP/HTP 不 fallback，需要从模型格式上满足后端：量化、算子可支持、layout 和 context 生成都要匹配。

## QNN DSP 失败原因

QNN DSP 是目前最明确的 HTP/CDSP 专项问题。旧的未改写 FP32 `.so` 不是“没进后端”，而是进入 HTP 初始化后模型校验失败。0703 后续已经证明 Pad+INT8 改写版完整模型可以执行，但精度弱于 SNPE DSP INT8。

当前失败特征：

```text
The SocModel doesn't support FP16
QnnBackend_validateOpConfig failed 3110
Failed to validate op _enc1_c1__prepad_Pad with error 0xc26
MODEL_GRAPH_OP_VALIDATION_ERROR
```

模型侧已知问题：

- ONNX 中存在多个 `Pad` 节点。
- 当前第一个明确失败点是 `/enc1/c1/_prepad/Pad`。
- 该 Pad 来自模型里的 ReflectionPad/reflect padding。
- 当前 QNN `.so` 仍是 FP32 路线。

判断：

- QNN DSP 失败更像“模型结构和精度格式不满足 HTP 后端要求”，不是单纯 runtime 字符串写错。
- 解决方向不应只继续试 `HTP/NPU/AIP/DSP_FIXED8_TF` 这类字符串。

最新 reflect Pad 最小复现：

| 最小模型 | QNN CPU | QNN GPU | QNN DSP |
| --- | --- | --- | --- |
| `reflect_pad_conv` | 通过 | 通过 | 失败在 `_pad_Pad` |
| `constant_pad_conv` | 通过 | 通过 | 失败在 `DepthWiseConv2d` |
| `builtin_pad_conv` | 通过 | 通过 | 失败在 `DepthWiseConv2d` |

这说明：

- reflect Pad 确实是 HTP/CDSP 兼容风险点。
- 仅替换 reflect Pad 还不够，因为最小 DepthWiseConv2d 在当前 FP32 QNN DSP 路线下也失败。
- 已继续拆普通 Conv、DepthWiseConv2d、INT8 量化三条最小复现，结果表明当前 QNN DSP 对 FP32 最小 Conv 图也会失败。

最新 Conv / INT8 最小专项：

| 最小模型 | QNN CPU | QNN GPU | QNN DSP | 结论 |
| --- | --- | --- | --- | --- |
| `plain_conv` | 通过 | 通过 | 失败在 `Conv2d` | FP32 普通 Conv 也不是 HTP 可执行图 |
| `depthwise_conv` | 通过 | 通过 | 失败在 `DepthWiseConv2d` | FP32 depthwise conv 同样不被当前 HTP 接受 |
| `qdq_int8_conv` | 失败 | 失败 | 失败 | 手写 Q/DQ 图不是可用 INT8 产物 |

真正的 converter 量化也已用最小 `plain_conv` 尝试：

- 已生成 4 个 float32 校准 raw 和 `input_list.txt`。
- 旧 QEMU `4.2.1` 下，`qnn-onnx-converter --input_list` 在 `qnn_quantizer.py` 的 `quantize` 阶段触发 QEMU `Illegal instruction`。
- 去掉 per-channel、改 `bias_bitwidth=8` 后，在旧 QEMU 下仍触发同样崩溃。
- 使用 QEMU `8.2.2` 后，最小 `plain_conv_int8` 已成功生成 QNN `.cpp/.bin/.json` 产物。

因此当前 HTP/CDSP 判断应更新为：

- QNN DSP 能进入 HTP/CDSP 后端，但 FP32/FP16 最小 Conv 图不能通过 HTP 校验。
- 当前本机必须固定到 QEMU `8.2.2` 后再推进量化；旧 QEMU `4.2.1` 不适合作为量化依据。
- 完整 `snpe-dlc-quantize` / `qairt-quantizer` 已在 QEMU `8.2.2` 条件下恢复，并生成 SNPE / QAIRT INT8 DLC。
- QNN DSP Pad+INT8 改写版完整 radio-map `.so` 已在 10 个真实 test 样本上跑通，平均 `90.18 ms`，RMSE `4.571 dB`，MAE `2.931 dB`。
- 当前 QNN DSP 专项的主要问题不再是“是否进入 HTP/CDSP”，而是“如何减少 Pad 改写和 INT8 QNN 图带来的精度损失”。

## HTP/CDSP 专项

HTP/CDSP 不建议作为当前部署主线的前置条件，但值得单独开专项。专项目标不是“马上跑通”，而是把失败原因缩小到可验证假设。

专项目录：

```text
modules/radio_map_qcs6490_deploy/docs/HTP_CDSP专项/
```

建议拆成 4 个实验：

1. 后端证据复核：确认 SNPE DSP、SNPE NPU、QNN DSP 分别加载了哪些库、打开了哪些设备节点。
2. 模型结构改造：导出一个去掉 reflect Pad 的小模型或替代模型，验证 QNN DSP 是否仍失败在 Pad。
3. INT8 量化路线：准备 representative input list，尝试生成真正面向 DSP/HTP 的量化 DLC 或 QNN context。
4. 转换工具稳定性：解决当前本机 Docker/QEMU 下量化工具容易 core 的问题，或把量化/上下文生成迁移到 x86_64 PC。

0703 后续已经完成第 2、3、4 项的首轮突破：QEMU 8.2.2 下量化恢复，Pad+INT8 QNN `.so` 可执行。后续专项应改为评估 padding 改写的精度影响，以及寻找 HTP 可接受、但更接近 reflect padding 行为的替代表达。

## 决策点

### 是否继续追 SNPE DSP/NPU

不建议作为近期主线。

理由：

- 当前 SNPE DSP/NPU 没有速度证据。
- 当前 DLC 不是量化模型。
- 课程文档只给 runtime 封装入口，没有给避免 fallback 的强制参数。
- 继续盲试 runtime 字符串收益很低。

### 是否继续追 QNN DSP

可以，但应作为专项。

理由：

- QNN DSP 确实进入 HTP/CDSP 初始化路径。
- 失败点集中在 FP16/SoC 支持和 `Pad` 节点校验。
- 它有明确可拆解的问题，不是完全黑盒。

### 是否改模型结构

如果目标是 HTP/CDSP，可能需要。

优先尝试：

- 把 reflect padding 改为 zero padding 或 Conv 内置 padding。
- 导出一个最小复现实验模型，只保留第一层 Pad + Conv。
- 如果最小模型仍失败，再集中查 QNN HTP 对 Pad scheme、DepthWiseConv2d 和 FP32/FP16 的支持。

代价：

- 改结构会影响论文模型输出，需要重新做 PyTorch/ONNX/SNPE/QNN 数值回归。
- 这属于部署优化，不应混入当前论文训练主线。

### 是否做 INT8 量化

如果目标是 DSP/HTP，应该做。

理由：

- SNPE DSP 日志选择了 `dsp_fixed8_tf`，这通常指向定点/量化路线。
- 当前 DLC 是 FP32，`Quantizer command: N/A`。
- HTP/DSP 对模型精度和算子支持比 GPU 更严格。

当前阻碍：

- 本机旧 QEMU `4.2.1` 下量化工具不稳定，当前已确认实际量化会触发 QEMU `Illegal instruction`。
- 本机 QEMU `8.2.2` 已让最小 `plain_conv_int8` 的 `qnn-onnx-converter --input_list` 量化转换成功，也已让完整 `snpe-dlc-quantize` / `qairt-quantizer` 恢复并生成 radio-map INT8 DLC。
- 需要准备 representative input list。
- 当前应固定 QEMU 8.2.2 作为本机转换前置条件。
- 后续需要确认 QNN context、`--backend DSP/HTP`、`--soc_model` 等参数组合。

### 是否换 x86_64 PC 做转换

这是现实备选。

当前本机可以完成 ONNX -> DLC、完整 INT8 DLC 和 QNN `.so` 的基础转换。旧 QEMU 4.2.1 下量化、context 生成、net-run 等工具不稳定；QEMU 8.2.2 已让最小 QNN INT8 转换和完整 SNPE/QAIRT DLC 量化通过。因此现在的优先级应是在本机固定 QEMU 8.2.2，x86_64 PC 只作为后备方案。

当前补充结论：

- 量化工具的 help 可运行，说明环境变量和工具路径可以配通。
- 使用真实 test 样本生成 raw input list 后，旧 QEMU 4.2.1 下 `snpe-dlc-quantize` 和 `qairt-quantizer` 的实际量化会触发 QEMU illegal instruction；QEMU 8.2.2 下已恢复。
- 最小 `plain_conv_int8` 在 QEMU 8.2.2 下已通过 `qnn-onnx-converter --input_list` 量化转换。
- 已按项目规则清理失败产生的 core 文件。
- QNN DSP Pad+INT8 改写版 full `.so` 已经可以执行，但精度弱于 SNPE DSP INT8。
- 因此当前不建议把 QNN HTP/CDSP full `.so` 作为部署主线，但可以继续把“QNN DSP 图改写精度优化和 context 生成”作为专项推进。

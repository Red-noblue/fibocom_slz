# 共享环境管理

## 目标

`env-bootstrap` 当前采用“描述文件分散、环境前缀集中”的模式：

- 项目依赖描述靠近模块
- 真实环境前缀集中到仓库根目录 `._envs/`
- 所有构建、运行、清理动作统一通过 `env-bootstrap`

## 为什么不用每个子项目一个 `.conda-env`

- 同一技术栈会重复下载和占用磁盘。
- IDE 和工具链容易扫描到多个大目录。
- 团队成员很难判断该进入哪个环境。
- CI、本地和迁移机器上的环境命名容易漂移。

## 当前约定

- 能复用时优先复用共享环境。
- 只有在 Python 版本、原生依赖或重型运行时冲突明显时才新建并行环境。
- 共享环境统一放在 `fibocom_slz/._envs/`。

推荐命名：

- `<domain>-py<major><minor>`
- 例如 `uav-modules-py310`

## 什么时候拆新环境

适合新建并行环境的典型情况：

- Python 主版本不同
- 需要 ROS / Gazebo / PX4 / AirSim / CUDA 等重型依赖
- 一组模块需要 GPU 栈，另一组只需要轻量数据分析栈
- 二进制库冲突导致共享环境维护成本过高

## 日常操作

查看共享环境：

```bash
./tools/env-bootstrap/bin/env-bootstrap list
```

构建指定环境：

```bash
./tools/env-bootstrap/bin/env-bootstrap sync \
  ./tools/env-bootstrap/manifests/uav-modules.env.sh
```

清理孤儿环境：

```bash
./tools/env-bootstrap/bin/env-bootstrap prune
./tools/env-bootstrap/bin/env-bootstrap prune --yes
```

## 边界

- `prune` 只清理 `._envs/` 下没有 manifest 引用的目录。
- `prune` 不会处理模块目录里的历史 `.conda-env`，这类迁移残留需要明确清理动作。
- lock 文件是平台相关资产，`linux-aarch64` 和 `linux-64` 不能直接互换。
